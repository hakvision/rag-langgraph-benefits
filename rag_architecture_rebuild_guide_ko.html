1. 이 저장소가 하는 일

이 프로젝트는 단순한 "RAG 답변기"가 아니라, 라운드별 구조 실험 + judged evaluation + HTML 보고를 한 세트로 운영하는 실험 프레임워크다.

실험 스크립트

round*.py 파일이 라운드별 파이프라인을 구현한다.

HTML 리포트

3. 최종 파이프라인 한 줄 흐름

source load→ chunk build→ classify_question→ question_type_router→ adaptive query plan→ multi-query retrieval→ fusion→ rerank→ judge_retrieval→ support stitch→ evidence distill→ citation answer→ judge_answer

질문이 쉬우면 짧은 경로로 끝내고, retrieval 품질이 낮거나 질문이 복합적일 때만 rescue/확장 분기를 연다.

4. 계층별 설명

질문 분류 계층

classify_question은 질문을 8b/14b lane으로 보낸다. detect_question_type은 이를 다시 simple_fact, abstract_why, multi_part로 나눠서 query 전략을 정한다.

이렇게 분리한 이유는 모델 크기 라우팅과 쿼리 형태 라우팅이 서로 다른 문제이기 때문이다.

Query planning 계층

build_adaptive_query_plan은 base query 위에 focus, rewrite, step_back, subquery, retrieval_rescue를 상황별로 덧붙인다.

단일 query로는 추상 질문과 복합 질문을 안정적으로 못 잡기 때문에 multi-query 설계를 택했다.

Retrieval / fusion 계층

retrieve는 lexical hit와 section prior로 각 청크를 스코어링한다. fuse_candidates는 각 query branch의 결과를 합쳐 source_queries, branch_hits, fused_score를 만든다.

Rerank / support 계층

rerank는 retrieval 후보를 answer-friendly ordering으로 다시 정렬한다. stitch_support_chunks는 judged evaluation에서 반드시 필요한 support chunk를 놓치지 않도록 보강한다.

Quality gate 계층

grade_search는 coverage와 top score를 기반으로 answer 또는 escalate_to_14b를 결정한다. retrieval 문제와 answer 문제를 분리하기 위해 answer judge와 독립시켰다.

Answer / judge 계층

distill_evidence가 핵심 근거를 짧게 압축하고, build_answer_text가 citation-aware template로 답을 조립한다. 마지막으로 evaluate_run이 keyword/support/quality 기반 rubric 점수를 계산한다.

5. round13 PDF branch가 추가한 것

소스 어댑터 교체

  • ensure_pdf()로 문서를 로컬 datasets 경로에 고정 저장
  • extract_pdf_text()로 실제 PDF에서 텍스트 추출
  • clean_text()로 머리글/쪽수/노이즈 제거
  • build_chunks()로 marker-based section split 수행

왜 marker split을 썼나

PDF는 HTML처럼 heading tree를 안정적으로 주지 않기 때문에, 이 branch는 범용성보다 재현 가능한 안정성을 우선했다. 따라서 문서 구조를 직접 읽어 SECTION_SPECS를 정의했다.

6. 점수 체계

55keyword score 가중치
25support chunk score 가중치
20retrieval quality score 가중치

이 평가 방식의 장점은 점수가 낮을 때 원인을 분리해서 볼 수 있다는 점이다. keyword가 빠졌는지, support chunk를 못 잡았는지, retrieval coverage가 낮았는지 바로 알 수 있다.

7. 다른 곳에서 다시 만들 때 추천 순서

  1. round1 수준의 baseline skeleton부터 구현

  3. question type router 추가
  4. adaptive query plan 추가
  5. fusion 추가
  6. retrieval rescue / support stitch 추가
  7. evidence distill / citation answer / refine 추가

이 순서가 좋은 이유는 각 계층의 기여도를 따로 디버깅하기 쉽기 때문이다.
