RAG × LangGraph

RAG에 LangGraph를 붙이면 뭐가 좋아지나?

한 줄 요약: "검색 기반 답변(RAG)"을 그냥 한 번 호출하는 수준에서 끝내지 않고, 여러 단계의 검색·검증·분기·승인까지 제어 가능한 워크플로우로 끌어올릴 수 있다.

상태 관리 조건 분기 병렬 처리 반복 검색 휴먼 인 더 루프 에이전트형 RAG

1. 먼저 결론

기본 RAG만 쓸 때

  • 질문 → 검색 → 문서 몇 개 가져오기 → 답변 생성
  • 구조가 단순해서 빠르게 만들기 좋음
  • 하지만 실패했을 때 왜 실패했는지 제어하기 어려움

LangGraph를 붙인 RAG

  • 질문 분류 → 검색 전략 선택 → 재검색 → 검증 → 요약 → 답변까지 흐름을 설계 가능
  • 복잡한 검색 시나리오를 안정적으로 다룸
  • 실서비스에서 필요한 통제·추적·승인을 넣기 좋음

2. 실제로 좋아지는 포인트 6가지

1) 검색 실패 시 재시도 로직

1차 검색 결과가 약하면 다른 쿼리로 다시 검색하거나, 키워드 검색 → 벡터 검색 → 하이브리드 검색 순서로 우회할 수 있다.

2) 질문 종류별 분기

정의형 질문, 비교형 질문, 최신성 필요한 질문, 내부 문서 질문을 구분해서 서로 다른 RAG 플로우로 보낼 수 있다.

3) 상태(State) 기반 누적 처리

질문, 검색 결과, 중간 요약, 검증 결과를 상태로 쌓아 두고 다음 노드가 이어받는다. 디버깅과 품질 개선이 쉬워진다.

4) 병렬 처리

여러 데이터 소스(사내 문서, 위키, 웹검색)를 동시에 조회한 뒤 합치는 구조가 가능하다. 속도와 회수율 모두 유리하다.

5) 휴먼 인 더 루프

민감한 답변이나 비용 큰 호출 전에는 사람 승인 단계를 넣을 수 있다. 기업 환경에서 특히 중요하다.

6) 반복형 에이전트 RAG

“정보가 부족하면 다시 찾고, 충돌하면 검증하고, 충분하면 답변” 같은 루프를 만들 수 있어 단발성 RAG보다 훨씬 똑똑해진다.

3. 어디에 특히 잘 맞나

Case 1사내 문서 QA

권한 있는 문서만 검색하고, 근거가 약하면 재검색 후 답변.

Case 2리서치 어시스턴트

사내 자료 + 웹 자료를 동시에 찾아 서로 충돌 여부까지 확인.

Case 3고객지원 봇

FAQ → 정책 문서 → 최신 공지 순으로 단계적 조회.

Case 4보고서 생성

검색 → 근거 추출 → 초안 작성 → 사람이 승인 → 최종 출력.

4. 구조 차이 한눈에 보기

항목 일반 RAG LangGraph 적용 RAG
흐름 거의 직선형 노드/엣지 기반 그래프형
실패 대응 한 번 실패하면 그대로 품질 저하 재검색, 우회, 검증, 반복 가능
데이터 소스 단일 소스 중심 복수 소스 병렬 조회 용이
운영 통제 제한적 상태 추적, 승인, 체크포인트 가능
적합한 수준 MVP / 단순 QA 프로덕션 / 복잡한 에이전트형 검색

5. 실무 관점에서 핵심 포인트

LangGraph의 본질은 “RAG 정확도 자체를 마법처럼 올리는 도구”라기보다, RAG 시스템을 운영 가능한 워크플로우로 바꾸는 프레임워크에 가깝다.

즉, 문서를 더 잘 찾게 해주는 건 임베딩/리트리버/리랭커의 역할이고, LangGraph는 그 여러 단계를 언제 어떤 조건에서 실행할지 조립해 주는 쪽에 강하다.

6. 그래서 언제 쓰면 좋나?

쓰는 게 좋은 경우

  • 검색 소스가 여러 개다
  • 재검색/검증/승인이 필요하다
  • 단순 챗봇보다 업무 플로우가 중요하다
  • 실패 원인을 추적해야 한다

굳이 안 써도 되는 경우

  • 문서 몇 개 붙여서 단순 QA만 만들면 된다
  • MVP를 아주 빨리 검증하는 단계다
  • 복잡한 분기 없이 “검색 후 답변”이면 충분하다

7. 한 줄 추천
