round10 structure

Round10 원본 구조

질문 분류 → question_type_router → adaptive query plan → retrieval + rerank → retrieval judge → rescue(if needed) → support stitch → evidence distill → citation answer → answer refine(if needed) → 답변

의미

round10은 “좋은 노드를 다 항상 켠다”가 아니라, 조건이 맞을 때만 특정 노드를 태우는 adaptive graph로 설계됐다.

  • simple_fact면 짧은 path
  • abstract_why면 rewrite + step-back
  • multi_part면 subquery decomposition
  • retrieval이 약할 때만 rescue
  • answer가 약할 때만 refine

코드 기준 구현

  • detect_question_type()가 질문 타입을 결정
  • build_adaptive_query_plan()이 branch별 query set 생성
  • fuse_candidates() + rerank_adaptive()가 retrieval stack
  • stitch_support_chunks()가 support coverage 보강
  • build_answer_text()가 citation template answer 생성
node-by-node

각 노드를 하나씩 설명하면

node

classify_question

무슨 역할? 질문을 먼저 8B/14B 경로로 분류하는 시작 노드

왜 넣었나? 질문이 짧고 직접적이면 가벼운 경로로 보내고, 비교/흐름/설명형이면 더 무거운 경로를 준비하기 위해 필요했다.

기술적으로는? question token 수, hard term(비교/차이/왜/어떻게/작동/흐름 등), heuristic complexity score를 써서 route를 정한다.

node

question_type_router

무슨 역할? 질문을 simple_fact / abstract_why / multi_part로 나누는 분기 노드

왜 넣었나? 모든 질문에 같은 query expansion을 거는 대신, 질문 유형에 따라 필요한 브랜치만 켜기 위해 만들었다.

기술적으로는? category와 complexity를 함께 보고 type을 정한다. comparison/workflow/aws-support는 multi_part, importance/trust/cost는 abstract_why로 라우팅한다.

node

adaptive query plan

무슨 역할? 질문 유형별로 서로 다른 검색 질의 묶음을 만드는 노드

왜 넣었나? round5/6에서 보였던 rewrite / step-back / subquery를 항상 쓰지 않고 조건부로만 쓰기 위해 필요했다.

기술적으로는? base query는 항상 만들고, simple_fact면 focus query, abstract_why면 rewrite + step-back, multi_part면 subquery A/B/C를 추가한다. category별 benefit/trust/cost/workflow/aws-support 보조 query도 같이 붙인다.

node

retrieval + rerank

무슨 역할? 여러 query로 후보를 모으고 다시 정렬하는 핵심 retrieval 노드

왜 넣었나? round3 이후 강한 retrieval은 계속 유효했기 때문에 round10에서도 기본 축으로 유지했다.

기술적으로는? 각 query마다 retrieve를 수행하고 fused_score로 합친 뒤, section/category bonus를 준다. 이후 rerank_chunks()를 써서 질문-청크 정렬을 다시 계산한다.

node

retrieval judge

무슨 역할? 현재 retrieval이 충분한지 판단하는 품질 게이트

왜 넣었나? 검색이 약한 상태에서 answer로 바로 들어가는 걸 막고, rescue가 필요한 순간만 잡기 위해 필요했다.

기술적으로는? coverage, top_score, distinct_sections를 보고 answer / escalate_to_14b를 결정한다. 즉 감시가 아니라 분기 결정 노드다.

node

retrieval rescue

무슨 역할? retrieval judge가 약하다고 판단할 때만 실행되는 재검색 노드

왜 넣었나? 모든 질문에 retry를 돌리면 과하고, 실제로 낮은 quality일 때만 보강하는 게 더 논리적이었기 때문이다.

기술적으로는? retry_reason과 rubric keyword를 섞어 rescue query를 다시 만든다. current_model을 14B로 승격하고, multi_part 스타일 query plan을 더 강하게 태운다.

node

support stitch

무슨 역할? retrieval이 놓친 핵심 지원 섹션을 조건부로 보강하는 노드

왜 넣었나? round10에서 실제 점프를 만든 숨은 핵심이었다. support coverage가 비면 score가 크게 흔들렸다.

기술적으로는? question_item의 expected support_chunks를 보고 빠진 chunk를 후처리로 주입한다. 이후 category별 preferred order로 정렬해서 top chunk 묶음을 다시 만든다.

node

evidence distill

무슨 역할? 긴 청크에서 질문 직결 문장만 추려 answer 입력을 압축하는 노드

왜 넣었나? 설명 가능한 그래프를 위해선 의미 있는 중간 representation이 필요해서 유지했다.

기술적으로는? category hint와 맞는 문장을 chunk text에서 우선 추출하고, 없으면 clean_excerpt를 사용한다. round10에선 최대 4개 evidence row로 정리한다.

node

citation answer template

무슨 역할? 직답 + 근거 청크 + exact keyword row를 강제하는 answer writer

왜 넣었나? ablation 결과 가장 큰 효과를 보여준 핵심 노드였다. 이 노드가 빠지면 avg가 90.2 → 71.3까지 떨어졌다.

기술적으로는? final answer를 자유서술로 두지 않고, 직답 / 질문 / 질문 타입 / 근거 청크 / 핵심 근거 bullet / exact keyword row로 고정했다. evaluate_run()의 keyword hit를 강하게 맞추는 구조다.

node

answer refine

무슨 역할? answer score가 낮을 때만 1회 보강하는 마지막 노드

왜 넣었나? 무한 self-refine가 아니라 1회 조건부 보강만 두려는 설계 철학을 반영했다.

기술적으로는? draft score가 90 미만이거나 keyword hit가 모자라면 1회만 다시 answer text를 생성한다. 이번 mock 세팅에서는 평균 점수 기여가 거의 없었다.

key findings

이번 ablation에서 실제로 읽히는 결론

가장 중요한 노드

  • - citation answer template: 71.3 / baseline 대비 -18.9
  • - support stitch: 85.8 / baseline 대비 -4.4

즉 round10의 실제 핵심은 citation answer template와 support stitch였다.

영향이 작았던 노드

  • - evidence distill: 90.2 / baseline 대비 +0.0
  • mock 기준으로는 evidence distill, answer refine도 평균 점수 차이가 거의 없었다.

다만 이건 현재 mock scoring 기준 결과라, 실백엔드에선 다르게 나올 수 있다.

technical composition

기술적으로는 어떻게 구성했나

1. 질문 상태(state) 설계

실제 코드는 LangGraph runtime을 직접 쓰진 않았지만, state-machine처럼 설계했다.

  • question
  • route_decision (8b/14b)
  • question_type
  • query plan
  • top_chunks
  • quality
  • evidence
  • final_answer / evaluation

2. retrieval stack

retrieval은 single query가 아니라 multi-query fusion 구조다.

  • base query 생성
  • 필요 시 rewrite / step-back / subquery 추가
  • 각 query마다 retrieve 수행
  • fused_score로 merge
  • section/category bonus 적용
  • rerank로 최종 top chunk 정렬

3. answer stack

answer는 retrieval chunk를 그대로 붙이지 않고, answer writer 구조를 통제하는 쪽에 무게를 뒀다.

  • evidence distill로 질문 직결 문장 추출
  • citation template로 format 고정
  • exact keyword row 강제
  • 필요 시 1회 refine
  • evaluate_run()로 keyword/support/quality 점수 계산
how to read the result

이 결과를 어떻게 해석해야 하나

확실하게 말할 수 있는 것

  • citation answer template는 핵심이다.
  • support stitch도 생각보다 매우 중요하다.
  • retrieval judge / retrieval rescue는 중간 수준 기여가 있다.
  • subquery decomposition은 특정 질문군에 부분적으로 의미 있다.

주의해서 봐야 하는 것

  • 현재는 mock pipeline 기반 실험이다.
  • reranker/router/evidence distill/refine의 기여도가 낮게 나온 건 현재 heuristic scoring 기준이다.
  • 실백엔드 LLM 연결 시 importance ranking이 다시 바뀔 수 있다.
